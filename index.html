1 · Context
Problem & Hypothesis
How do we turn “cute screenshot of an outfit” into a cart without friction?

Problem.

  • “Shop the look” modules often feel gimmicky – low trust and low match quality.
  • Creators post fit pics; viewers screenshot and bounce to search manually.
  • Retailers can’t measure whether UGC → carts without heavy integration.

Hypothesis.

  • If we let users upload an outfit and instantly see 3 similar products, add-to-cart will increase vs a static “Shop the look” tile.
  • Latency is critical: under ~10 seconds still feels “magic,” above that feels broken.

Scope of this MVP.

  • Single-page static app (no server) using precomputed HSV histograms.
  • Top-3 match list only – no full PDP experience.
  • Focus on: upload reliability, perceived match quality, early funnel behavior.

Key facts.

  • Experiment flag: shoplens.enabled
  • Tickets: SHOP-101, SHOP-118 · PRs #902, #917, #921.

4 · Insights
User Research + Data Schema
Why the experience works and how it’s instrumented.

User research (n = 30).

  • “Under 10 seconds feels instant” → we kept p50 ≈ 8s.
  • “Do these really match?” → added color/overall labels and simple copy.
  • Size confirmation as auto-prefill from history (editable) reduced friction.
  • “Find similar” copy beat “Shop the look” by ~6% CTR in concept tests.

How it works (simple version).

  1. User uploads a screenshot of an outfit (client-side only).
  2. We compute a tiny HSV histogram in the browser.
  3. We compare against precomputed histograms in products.json and show the top-3.

Data & schema.

AA test (n≈5k) showed no SRM (p≈.53). Late events <4% (24h watermark, UTC normalization). Events: upload_started, upload_ok, match_ready, pdp_view, add_to_cart. TABLE shoplens_events ( event_time_utc TIMESTAMP, user_id STRING, session_id STRING, event_name STRING, product_id STRING, match_score DOUBLE, bucket STRING, -- e.g. 'price_sensitive' experiment_key STRING, variant STRING -- 'A' | 'B' );

6 · Risks & Architecture
Risks & Mitigations · Architecture Doodle

Risks & mitigations.

  • Content safety. Filter v2 + reviewer queue for flagged uploads.
  • Model drift. Monthly retrain; shadow eval on holdout set.
  • Latency spikes. Warm cache, eager transforms, queue back-pressure.
  • Cold catalog. Fallback to “similar but cheaper” or “same brand” matches.

Architecture (high-level).

+-----------+ +---------+ +---------+ | Client | → | API | → | Images | | (browser) | | layer | | store | +-----------+ +---------+ +---------+ ↓ ↓ ↑ HSV histo Match service Offline jobs (local) / scoring API building histograms

Current MVP runs entirely client-side (no API); this is the target architecture if we plug into a live catalog.

7 · Demo
ShopLens Demo Flow
What a typical user sees – 3-step story you can talk through live.
  1. Upload an outfit. User drags in a screenshot or taps “Choose file” – progress bar starts immediately.
  2. Get top-3 matches. After ~8 seconds, three cards appear, each showing product photo, price, and “because of: color / silhouette / vibe” explanation.
  3. Tap to shop. User taps a card → one-tap PDP → adds to cart with pre-selected size.
For live talks / interviews you can keep this slide up, run the live demo in another tab, and tie every step back to the numbers from earlier slides.

Notes for presenter.
